# TeamWicked Docs Design System

## 디자인 방향

문서 사이트는 `/home/ubuntu/teamwicked` 공개 사이트의 시각 언어를 정보 밀도가 높은 문서 환경에 맞게 적용합니다. 기준 콘셉트는 `design/concepts/docs-desktop.png`, `docs-mobile.png`, `admin-editor.png`이며 실제 데이터가 없는 메뉴나 통계는 콘셉트 재현을 위해 만들지 않습니다.

## 토큰과 사용 규칙

- **배경**: 밝은 모드의 true white를 기본으로 사용하고 표면은 얇은 회색 경계와 부드러운 그림자로 구분합니다.
- **브랜드 색**: 짙은 올리브는 구조·선택 상태, 라임은 활성 표시와 핵심 행동, 녹색은 성공 피드백에 사용합니다.
- **타이포그래피**: TeamWicked 실제 UI와 같은 `next/font/google`의 Geist를 영문 제목·본문·UI 전체에 사용하고, 코드는 Geist Mono를 사용합니다. `latin`에 없는 한글은 시스템 한글 폴백으로 표시합니다.
- **형태**: 헤더, 검색, 카드, 편집기 패널은 큰 radius를 사용합니다. 코드·메타 컨트롤은 더 작은 radius로 정보 계층을 구분합니다.
- **레이아웃**: 데스크톱은 좌측 문서 트리·중앙 본문·우측 목차의 3단 구조입니다. 모바일은 본문을 우선하고 메뉴와 목차를 시트로 이동하며 하단 도크에 핵심 동작을 둡니다.
- **다크 모드**: 단순 반전 대신 표면·본문·경계·코드 배경 토큰을 각각 바꿔 대비와 계층을 보존합니다.
- **테마 전환 모션**: TeamWicked 원본과 같은 520ms 원형 View Transition을 사용해 누른 토글 위치에서 새 테마가 화면 전체로 퍼집니다. View Transition 미지원 브라우저와 `prefers-reduced-motion` 환경에서는 즉시 전환합니다.

## 컴포넌트 원칙

- 문서 내비게이션은 실제 SQLite의 카테고리와 공개 문서만 표시합니다.
- 검색 결과는 제목·설명·카테고리를 먼저 보여 주고 키보드 접근을 지원합니다.
- 본문 줄 길이와 행간을 읽기 중심으로 제한하며 목차는 현재 제목을 명확히 표시합니다.
- 관리자 화면도 공개 사이트와 같은 토큰을 쓰되, 입력·저장 상태·위험 동작의 구분을 우선합니다.
- 애니메이션은 검색·시트·상태 전환을 설명하는 짧은 동작에만 사용하고 `prefers-reduced-motion`을 존중합니다.

## 시각 QA 기준

다음 항목을 데스크톱 1440×1000과 모바일 390×844에서 비교합니다.

1. 흰색 라운드 헤더의 높이·간격·브랜드 배치
2. 올리브 사이드바와 라임 활성 표시
3. 제목·본문의 폰트 계층과 true-white 바탕
4. 데스크톱 3단 구조와 모바일 하단 도크 전환
5. 검색, 목차, 코드 블록, 관리자 우측 속성 패널의 표면 일관성

콘셉트의 샘플 문구와 가상 메뉴는 시각적 방향만 제시합니다. 운영 화면은 DB의 실제 문서 제목·내용·카테고리를 유지하며, 그로 인한 줄바꿈이나 메뉴 수 차이는 의도된 데이터 기반 차이입니다.

[Decision Log]
- 목적과 의도: TeamWicked 브랜드를 그대로 복제하는 데 그치지 않고 긴 문서를 읽고 편집하기 좋은 일관된 UI 규칙으로 만든다.
- 기존 구현 및 제약 조건: 실제 문서 데이터의 길이와 카테고리 수는 고정되어 있지 않고, 모바일에서도 모든 기능에 접근할 수 있어야 한다.
- 검토한 주요 대안: 범용 GitBook 스타일, 공개 사이트 CSS 직접 복사, 브랜드 토큰을 문서 전용 컴포넌트로 재구성.
- 선택한 방식: 로고·폰트·핵심 색·형태는 공유하고 정보 구조와 반응형 동선은 문서 사용성에 맞게 재구성한다.
- 다른 대안 대신 이 방식을 선택한 이유: 브랜드 정체성을 유지하면서도 3단 문서 탐색과 관리자 편집에 필요한 밀도·접근성을 확보할 수 있다.
- 장점, 단점 및 영향: 사이트 간 인상이 통일되고 유지보수 기준이 명확해지지만 TeamWicked 원본 토큰이 변경되면 이 문서와 `globals.css`를 함께 검토해야 한다.

## 테마 전환 결정 기록

[Decision Log]
- 목적과 의도: TeamWicked 사이트와 Docs 사이의 라이트·다크 전환 경험을 동일하게 만든다.
- 기존 구현 및 제약 조건: Docs는 테마 클래스를 즉시 교체해 시각적 연결이 없었고, 헤더·모바일 도크·메뉴에 서로 다른 크기의 토글이 존재한다.
- 검토한 주요 대안: CSS 색상 transition, 화면 중앙 fade, TeamWicked의 클릭 위치 기반 View Transition.
- 선택한 방식: 모든 Docs 토글이 공유하는 520ms 원형 View Transition을 사용한다. Android Chrome의 첫 View Transition 스냅샷에서 절대 `px` 좌표가 다른 스케일로 해석될 수 있으므로, 눌린 버튼의 중심을 visual viewport 대비 백분율로 변환해 원점으로 사용한다.
- 다른 대안 대신 이 방식을 선택한 이유: TeamWicked 원본의 모션을 그대로 유지하면서 토글 위치와 새 테마 사이의 인과관계를 가장 명확히 보여 준다.
- 장점, 단점 및 영향: DPR, 모바일 주소창, 첫 스냅샷 크기와 무관하게 원형이 항상 누른 버튼 중심에서 시작한다. View Transition 미지원 브라우저에서는 즉시 전환하며, 모션 감소 환경에서도 접근성을 위해 애니메이션을 생략한다. 완료된 reveal animation은 명시적으로 취소해 다음 전환에 잔류하지 않게 한다.
